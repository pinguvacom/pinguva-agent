# Локальная интеграция Bitrix24

Этот документ объясняет, где взять секретный входящий вебхук Bitrix24, зачем он нужен и куда его вставлять при настройке Pinguva.

## Что вводится в UI Pinguva

В UI Pinguva вводится только публичный адрес портала Bitrix24:

```text
b24.example.kz
```

Pinguva использует его, чтобы сгенерировать команду для сервера Bitrix24. Секретный вебхук в UI Pinguva не вводится.

Подключение использует тот же Linux-агент Pinguva, который уже установлен на сервере Bitrix24. Отдельный агент не устанавливается. Для локальной диагностики нагрузки команда подключения включает короткий системный таймер: он собирает только обезличенную техническую сводку, а основной агент передаёт её в Pinguva.

## Что вводится в терминале

После запуска команды агент спросит:

```text
Bitrix24 webhook URL / Входящий webhook Bitrix24 (hidden input, not the portal URL):
```

Сюда нужно вставить входящий вебхук, созданный внутри Bitrix24. Это секрет. Он вводится скрыто, не отображается на экране и сохраняется только локально на сервере клиента.

## Как создать входящий вебхук в Bitrix24

1. Войдите в Bitrix24 под пользователем, от имени которого можно безопасно выполнять REST-проверки.
2. Лучше использовать отдельного технического пользователя с минимальными правами, а не личный аккаунт администратора.
3. Откройте в Bitrix24 раздел `Приложения`.
4. Перейдите в `Разработчикам`.
5. Откройте `Готовые сценарии`.
6. Выберите `Другое`.
7. Выберите `Входящий вебхук`.
8. Задайте понятное название, например `Pinguva REST monitoring`.
9. В блоке прав выдайте только те права, которые нужны выбранным REST-профилям Pinguva.
10. Сохраните вебхук.
11. Скопируйте URL из генератора запросов.

Официальная документация Bitrix24 описывает этот путь как `Приложения > Разработчикам > Готовые сценарии > Другое > Входящий вебхук`.

## Какие права выбрать

Для стандартного набора профилей выдайте входящему webhook права:

- `Пользователи: краткие данные (user_brief)` — для базовой проверки `user.current`, проверки выданных REST-прав через `scope` и опциональной проверки `user.get`;
- `CRM` — если включены `crm_deals`, `crm_leads`, `crm_contacts` или `crm_statuses`.

`scope` не является отдельным правом в Bitrix24: этот профиль только проверяет
REST-права, которые уже выданы webhook.

Для отдельного технического пользователя настройте только чтение сделок, лидов
и контактов. Не используйте личный аккаунт администратора и не выдавайте
вебхуку больше прав, чем нужно. Если CRM-профили не используются, не включайте
CRM-доступ.

Профиль `method_discovery` выключен по умолчанию. При ручном включении он
проверяет доступность метода `user.get` через `method.get`.

## Что именно копировать

В Bitrix24 URL вебхука выглядит примерно так:

```text
https://b24.example.kz/rest/1/secret_code/profile/
```

или как полный пример вызова метода:

```text
https://b24.example.kz/rest/1/secret_code/user.current.json
```

Для Pinguva можно вставить полный URL из генератора запросов. Агент возьмёт из него только базовую часть вебхука:

```text
https://b24.example.kz/rest/1/secret_code
```

Не добавляйте этот URL в UI Pinguva, командную строку, тикеты поддержки или логи.

## Куда вставить вебхук

Запустите короткую команду, которую Pinguva сгенерировала в карточке Bitrix24:

```bash
curl -fsSL "https://monit.pinguva.com/install/bitrix24.sh" | sudo bash -s -- --base-url "https://b24.example.kz"
```

Команда обновит существующий Linux-агент Pinguva, запустит локальную настройку Bitrix24, включит минутный таймер локальной диагностики и после сохранения перезапустит тот же сервис `pinguva-agent`. После запуска появится скрытый ввод:

```text
Bitrix24 webhook URL / Входящий webhook Bitrix24 (hidden input, not the portal URL):
```

Вставьте вебхук и нажмите `Enter`. Символы не будут отображаться. Это нормально.

## Где хранится секрет

Вебхук сохраняется только на сервере клиента:

```text
/etc/pinguva-agent/bitrix24.json
```

Файл создаётся с ограниченными правами доступа. В Pinguva отправляется только техническая сводка проверки: статус, задержка, выбранный REST-метод, безопасный счётчик и короткая ошибка после удаления секретов.

## Локальная диагностика нагрузки

Начиная с версии агента `0.2.6` команда подключения также включает
`pinguva-bitrix24-diagnostics.timer`. Раз в минуту он локально читает только
ограниченное окно стандартного Nginx/Apache access log и технический статус
MySQL. Это помогает понять, какой маршрут приложения создаёт нагрузку, не
передавая журналы и CRM-данные в Pinguva.

В интерфейсе Pinguva будут видны:

- количество динамических запросов и `5xx` за последние 15 минут;
- самые активные нормализованные пути, например `/api/accruedpoints`;
- источники трафика с маскированным последним сегментом IP;
- `Threads_running`, количество активных запросов и максимальная длительность
  текущего запроса MySQL;
- фиксированные технические категории запросов, например поиск CRM-контактов
  без учёта регистра.

Начиная с версии агента `0.2.9`, задача сначала использует `/root/.my.cnf`,
только если это обычный файл `root:root` со строго установленными правами
`0600`. Это обычный сценарий для коробочного Bitrix24: пароль остаётся на
сервере и не попадает в аргументы команд, журналы или Pinguva. Небезопасный
файл агент не изменяет и вместо него пробует стандартный локальный socket MySQL.

Для MySQL и MariaDB агент сначала использует
`performance_schema.global_status`, затем `SHOW GLOBAL STATUS`, а последним
вариантом - `information_schema.GLOBAL_STATUS`. Состояние потоков и список
активных запросов собираются отдельно: `0` активных запросов и `0` секунд
максимальной длительности означают спокойный снимок, а не ошибку.

Если access log или MySQL недоступны локальной задаче, REST-проверки Bitrix24
не останавливаются. В карточке сервера Pinguva отдельно покажет недоступную
сводку.

## Обновление уже подключенного Bitrix24

Для уже настроенного Bitrix24 не нужно повторно создавать webhook и запускать
подключение заново. В карточке этого сервера Pinguva покажет, что доступна
диагностика Bitrix24. Скопируйте и выполните обычную команду обновления агента.

Начиная с версии `0.2.7` она делает только добавочные действия:

- обновляет бинарник агента;
- только при существующем `/etc/pinguva-agent/bitrix24.json` включает локальную
  диагностику и `pinguva-bitrix24-diagnostics.timer`;
- не меняет webhook, REST-профили и нестандартный путь access log;
- запускает первый локальный снимок.

Через одну-две минуты в карточке появятся агрегаты маршрутов, `5xx`, источников
и MySQL. В разделе диагностики можно выбрать до десяти уже обнаруженных важных
маршрутов. Pinguva хранит только нормализованные пути, а агент получает их в
ответе на собственный исходящий отчёт. Это не создаёт новых REST-запросов и не
меняет настройки Bitrix24.

Если access log находится не в стандартном системном пути, повторите команду
подключения с локальным путём журнала. Сам путь и содержимое журнала в Pinguva
не отправляются:

```bash
curl -fsSL "https://monit.pinguva.com/install/bitrix24.sh" | sudo bash -s -- \
  --base-url "https://b24.example.kz" \
  --access-log "/srv/bitrix/logs/nginx-access.log"
```

## Что не отправляется в Pinguva

Pinguva не получает:

- входящий вебхук Bitrix24;
- тела запросов и ответов;
- CRM-записи;
- названия сделок;
- телефоны;
- email;
- комментарии;
- пользовательские поля;
- файлы и вложения.
- сырые строки access log;
- параметры URL, cookies и заголовки авторизации;
- текст SQL и значения SQL-запросов.

## Проверка после настройки

```bash
sudo pinguva-agent bitrix24 status
sudo journalctl -u pinguva-agent -n 50 --no-pager
sudo systemctl status pinguva-bitrix24-diagnostics.timer --no-pager
sudo journalctl -u pinguva-bitrix24-diagnostics.service -n 50 --no-pager
```

Чтобы убедиться, что показатели отправлены, не раскрывая пароль:

```bash
sudo stat -c '%a %U:%G %F' /root/.my.cnf
sudo systemctl start pinguva-bitrix24-diagnostics.service
sudo journalctl -u pinguva-bitrix24-diagnostics.service -n 50 --no-pager | grep 'Bitrix24 MySQL diagnostics'
```

Ожидаемые права файла: `600 root root regular file`. Успешная строка журнала
содержит `connection=defaults_extra_file`, источник статуса и `status=ok`.
Следующий обычный отчёт агента передаст агрегаты в Pinguva в пределах одной-двух
минут. Никогда не выводите содержимое `/root/.my.cnf` в журнал или в обращение
в поддержку.

Если команда `bitrix24` неизвестна, обновите агент до версии `0.2.7` или новее.

## Если пользователь, создавший вебхук, изменился

REST-вызовы через входящий вебхук выполняются с правами пользователя, который его создал. Если этого пользователя отключили, удалили или урезали ему права, проверка Bitrix24 может перестать работать. В этом случае создайте новый вебхук и повторите настройку агента.
